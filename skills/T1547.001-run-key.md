# T1547.001 — Registry Run Key / Boot Autostart

## Tool sequence
1. `find /mnt/host -name 'NTUSER.DAT' 2>/dev/null | head -5` — locate user hives.
2. `find /mnt/host -path '*/System32/config/SOFTWARE' 2>/dev/null` — locate system hive.
3. `strings /mnt/host/Windows/System32/config/SOFTWARE 2>/dev/null | grep -iA2 -B2 'Run'` — scan for Run key entries.
4. `strings <NTUSER.DAT path> 2>/dev/null | grep -iA2 -B2 'CurrentVersion\\Run'` — user-level Run keys.

## CONFIRMED when
- Run key entry found containing a path to a known-malicious binary (httppump, spinlock, HYDRAKATZ, a.exe).
- Run key entry points to an unusual path (Recycle Bin, Temp, AppData) for executable persistence.
- Cross-check: the referenced binary must physically exist on disk.

## REFUTED when
- No Run key entries containing attacker tool names after checking both NTUSER.DAT (all users) and SOFTWARE hive.
- Run key entries present but all point to known-legitimate software (Windows Defender, Office, etc.).
- **Expected result for this campaign:** T1547.001 is consistently refuted — persistence is achieved
  via service installation (T1543.003) rather than Run keys.

## Case context
T1547.001 is refuted on all four campaign hosts. The Memory Agent flags it because process names sometimes
appear in registry string output. The Auditor must check the actual Run key values, not just grep for
process names in hive strings.
