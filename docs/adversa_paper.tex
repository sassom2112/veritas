\title{The Forensic Auditor: Architectural Isolation Between Detection and
       Verification Prevents LLM Hallucination in Autonomous DFIR}

\author{Scott Assom}
\affiliation{\institution{SANS FIND EVIL! Hackathon 2026}}
\email{sassom2112@gmail.com}

\begin{abstract}
LLM-based forensic agents hallucinate because the same agent that proposes
a finding also evaluates it. Prompt guardrails produce skeptical-sounding
output; they do not produce verified findings. VERITAS addresses this
structurally: a second agent---the Forensic Auditor---receives only a list
of detected technique identifiers and nothing else from the first agent's
session. Its mandate is refutation. A \textsc{confirmed} verdict requires a
positive return value from a real forensic tool call against physical disk
or memory evidence. Model confidence produces neither \textsc{confirmed} nor
\textsc{refuted}. This is a data-flow property, not a prompt property, and it
is verifiable by reading the code.

The system was validated on four hosts from a coordinated APT intrusion,
including three victim machines and a confirmed C2 relay node. Across all four
hosts the Forensic Auditor refuted exactly four techniques per host---sixteen
total---while confirming thirty-two from forty-eight detected. Every confirmed
finding is independently reproducible from an append-only audit log with one
shell command on the same mounted image. The consistent four-refutal pattern
across host types and tool families demonstrates that the architecture enforces
the same verification standard regardless of what it is pointed at.

Bensalem et al.~\cite{bensalem2026} argue that enforcing LLM agent safety
within a single abstraction layer is structurally insufficient and call multi-agent
settings ``the most important unfinished business in LLM agent runtime assurance.''
Where their framework derives probabilistic safety bounds for environmental validity,
VERITAS enforces this property categorically: \textsc{confirmed} cannot be issued
without a positive tool return value. No probability estimate is needed because
the property is structurally enforced by the data flow.
\end{abstract}

\keywords{digital forensics, MITRE ATT\&CK, DFIR automation, multi-agent systems,
hallucination, architectural enforcement, LLM agents, forensic admissibility}

%% ── 1. Introduction ──────────────────────────────────────────────────────────
\section{Introduction}

LLM-based forensic agents hallucinate. This is not a model quality problem---it
is a structural property of any system where the same agent that generates a
finding also evaluates it. Ask a generative model whether credential dumping
occurred on a disk image and it will find something that looks like credential
dumping, because it is trained to be helpful and because it has seen enough
incident reports to pattern-match convincingly. Prompt guardrails produce
skeptical-sounding output. They do not produce verified findings. A model
instructed to ``only report confirmed evidence'' and a model architecturally
prevented from confirming without a positive tool return are not equivalent.

The leading autonomous DFIR platform, Valhuntir~\cite{valhuntir}, reached the
same conclusion honestly: ``If you just tell it to Find Evil, it will
hallucinate.'' Their solution preserves a senior human examiner in the
verification loop. This is correct for general-purpose LLMs operating without
architectural constraints. It is also a deliberate concession that the
verification problem remains open.

VERITAS builds on Valhuntir's investigation layer and adds the missing piece: an
independent verification agent that sits between the investigation output and the
final report. The Triage Agent investigates---calling real SIFT tools, reading
real disk artifacts---exactly as Valhuntir's approach does. The Forensic Auditor
verifies. The human examiner reviews confirmed, artifact-backed findings instead
of raw LLM output. The verification problem that Valhuntir left open by design
has an architectural answer.

\textbf{Contributions.} We make three concrete contributions:
\begin{enumerate}
\item A dual-agent architecture in which the finding-proposal agent and the
finding-verification agent share zero session state, enforcing that
\textsc{confirmed} requires a positive forensic tool return
(\cref{sec:auditor}).
\item A purpose-built MCP security boundary with four independently motivated
gates that make evidence modification and command injection structurally
impossible before any subprocess executes (\cref{sec:security}).
\item Empirical validation across four hosts---three victim machines spanning
two tool families and one C2 relay node---demonstrating consistent Auditor
behavior: exactly four memory-only signals correctly refuted per host
(\cref{sec:evaluation}).
\end{enumerate}

%% ── 2. Related Work ──────────────────────────────────────────────────────────
\section{Related Work}
\label{sec:related}

\textbf{Autonomous DFIR agents.}
Valhuntir~\cite{valhuntir} demonstrated that connecting LLM agents to the SANS
SIFT Workstation via MCP enables broad automated investigation across disk
images, memory captures, and network artifacts. Their honest finding---that
autonomous Find Evil hallucinates without human direction---motivates the present
work. VERITAS extends their investigation layer with an architectural verification
layer rather than replacing it.

\textbf{Multi-agent debate and adversarial verification.}
Du et al.~\cite{du2023debate} showed that having multiple LLM instances debate a
claim improves factual accuracy over single-agent reasoning. VERITAS applies a
stronger version of this principle to a higher-stakes property: not factual
improvement but forensic verifiability. The Auditor does not debate the Triage
Agent---it independently calls real forensic tools and requires a positive return
value before issuing \textsc{confirmed}. Disagreement between agents is not the
goal; structural impossibility of hallucinated confirmation is.

\textbf{Formal agent safety verification.}
Fang~\cite{fang2026} proposes AgentVerify, a model-checking framework that
verifies agent safety properties using Linear Temporal Logic over observable
control flow. In evaluation across 15 diverse scenarios, post-hoc behavioral
analysis achieved 86.67\% verification accuracy, outperforming both monolithic
contract baselines (80.00\%) and runtime monitoring without temporal logic
(46.67\%). AgentVerify operates post-hoc on observable control flow and is
evaluated on designed scenarios. VERITAS achieves a stronger property
proactively: the architecture makes it structurally impossible to produce a
confirmed hallucination before the finding reaches the report, not merely
detectable afterward.

\textbf{Multi-layer agent safety architecture.}
Bensalem et al.~\cite{bensalem2026} argue that three safety dimensions---semantic
intent compliance, environmental validity, and dynamical feasibility---each
depend on information available at different execution stages, making
single-layer guardrails categorically insufficient. They derive compositional
probabilistic safety bounds via the chain rule and identify multi-agent settings
as ``the most important unfinished business in LLM agent runtime assurance.''
VERITAS implements all three layers---system prompts (Layer~1), the Forensic
Auditor (Layer~2), and the MCP validator (Layer~3)---and provides a stronger
guarantee than their probabilistic framework admits for Layer~2: the data flow
makes false \textsc{confirmed} verdicts structurally impossible rather than
merely improbable.

\textbf{Forensic admissibility of AI-generated evidence.}
TrueAllele DNA mixture interpretation and COMPAS recidivism scoring have both
been challenged in court on the grounds that model internals cannot be inspected
or reproduced~\cite{aronson2010}. The emerging DFIR equivalent will challenge
neural-network-generated technique flags on identical grounds. VERITAS confirmed
findings are designed to survive this scrutiny: every finding is a forensic tool
call and its output, not a model inference. A second examiner can reproduce any
confirmed finding from the audit log without the AI.

\subsection{Detection Layer (Proof of Concept)}
\label{sec:detection}

The detection layer exists to generate candidates for the Auditor to challenge.
It is not the contribution of this paper.

\textbf{Phase 1 --- Deterministic sweep} ($<$60\,s, no LLM). Approximately
25~SIFT commands run against the mounted image and score against corpus-calibrated
log-odds weights derived from 800+ labeled malware samples (MalwareBazaar,
HybridAnalysis):
\[
  w(s) = \text{normalize}\!\left(\log_2 \frac{p_\text{malware}(s) + 0.05}
                                              {p_\text{benign}(s)  + 0.05}\right)
\]
Every weight traces to a source SHA-256. The sweep is invariant across
investigations---nothing from a previous case affects the baseline. This
phase is used to decide whether a full agentic run is warranted.

\textbf{Phase 2 --- Agentic investigation} (75-call Claude budget). A
Claude-powered loop reads event logs, prefetch binaries, SAM/SECURITY hives,
shellbags, LNK files, and binary strings. \emph{Critically, the agent receives
raw artifact strings only---no Phase~1 scores, no technique labels.} This is an
architectural decision. Passing triage scores downstream caused measurable
anchoring bias: the LLM confirmed what it was told to look for. Structural
decoupling fixed it.

\textbf{Memory analysis} (parallel). Volatility~3 runs concurrently against the
raw memory image, contributing process injection signals, credential hashes, and
VAD anomalies invisible on disk.

On the four evaluated hosts, Phase~1 surfaced 2 of 48 total candidate techniques.
Phase~2 and memory analysis surfaced the remaining 46. The detection layer is a
starting point. The Auditor is what produces trusted output.

\subsection{The Forensic Auditor}
\label{sec:auditor}

After investigation completes, the Forensic Auditor challenges every detected
technique in parallel via \texttt{asyncio.gather}. Each technique runs in its own
isolated MCP session with no shared state from the Triage Agent.

\textbf{Input.} A JSON list of technique identifiers. Nothing else. No reasoning
traces, no confidence scores, no raw session context from the Triage Agent.

\textbf{Mandate.} Assume every finding is false until the filesystem proves
otherwise.

\textbf{Budget.} Up to 5 challenge rounds, 2 independent tool calls per round.

\textbf{Verdict logic.}
\begin{itemize}
  \item \textsc{confirmed}: the Auditor exhausts its budget without finding a
    contradiction, requiring at least one positive tool return value.
  \item \textsc{refuted}: positive evidence of absence---the artifact is not
    where the technique would require it to be.
  \item \textsc{inconclusive}: budget exhausted, evidence insufficient for
    either verdict. The system degrades gracefully: timeout returns
    \textsc{inconclusive}, not \textsc{confirmed}.
\end{itemize}

\textbf{The structural guarantee.} \textsc{confirmed} cannot be produced by
model confidence, a persuasive reasoning chain, a plausible-sounding argument,
or an attacker-controlled log entry. The data flow enforces this: the Auditor
receives technique IDs and calls forensic tools. Positive tool output is the
only path to \textsc{confirmed}. This is verifiable by reading
\texttt{auditor\_agent.py}; no evaluation is required to establish the property.

\subsection{IOC Campaign Propagation}
\label{sec:ioc}

Confirmed artifacts from one host propagate to subsequent investigations via a
dedicated command-line switch. The tool extracts verified IOCs into a structured
JSON file and injects them into the next target's deterministic sweep---no LLMs,
no API calls. This passes hard, structured evidence between hosts without
exposing the downstream investigation agent to the prior case's raw context or
confidence levels, preventing cross-contamination of the verification chain.

%% ── 4. MCP Security Boundary ─────────────────────────────────────────────────
\section{MCP Security Boundary}
\label{sec:security}

Every forensic action flows through one MCP primitive: \texttt{run\_terminal\_command}.
Four gates execute in Python before any \texttt{subprocess.run} call:

\textbf{Gate~1 --- Argument injection prevention.} 22 hard-blocked tokens:
destructive operations (\texttt{shred}, \texttt{mkfs}, \texttt{fdisk}), exfiltration
tools (\texttt{curl}, \texttt{wget}, \texttt{nc}), privilege escalation
(\texttt{sudo}, \texttt{pkexec}), and command substitution operators
(\texttt{\$(}, backtick, \texttt{\$\{}). Command substitution is blocked
because an attacker who controls log content or filesystem artifacts can inject
a second command as an argument to an otherwise legitimate forensic invocation.

\textbf{Gate~2 --- Deny-all binary enforcement.} 53-binary SIFT allowlist.
Every binary not on the list is rejected unconditionally. \texttt{sed} is
excluded despite appearing on SIFT because its \texttt{-e} flag passes the
pattern space to the shell, creating an injection surface.

\textbf{Gate~3 --- Quote-aware pipeline parser.} Each pipe segment is validated
independently after tracking single-quoted substrings. A na\"{i}ve split on
\texttt{|} would reject \texttt{grep -iE '(http|https|ftp)'} because
\texttt{https} and \texttt{ftp} are not allowlisted binaries. The parser
correctly identifies \texttt{|} inside single quotes as argument content, not
a pipeline separator.

\textbf{Gate~4 --- Write-target guard.} All \texttt{>}, \texttt{>>}, and
\texttt{tee} targets are resolved via \texttt{os.path.realpath()} and must
resolve inside \texttt{reports/}. Symlink traversal and \texttt{../} injection
fail at the path arithmetic level, not at a string-match level.

Every command is atomically appended to \texttt{audit\_log.jsonl} via
\texttt{os.open + os.write} \emph{before} \texttt{subprocess.run} is called.
Blocked commands log their \texttt{blocked\_reason}. The audit log cannot be
overwritten through a tool call. Evidence modification is structurally impossible,
not prompt-dependent.

Gates~1 and~2 address distinct attack vectors: binary substitution (an
unapproved executable) and argument injection (an approved executable with
malicious arguments). Both must be defended independently; Gate~2 alone does not
prevent a command like \texttt{grep -e \$(curl attacker.com) file}.

\textbf{The four-refutal pattern.} Every host produced exactly four refuted
techniques: T1071.001, T1134, T1547.001, and T1574. These are the same four
techniques across all four hosts, each flagged by memory analysis and each
lacking disk corroboration. The consistency of this pattern across three victim
machines with different tool families (httppump on nfury/tdungan,

Auditor applies the same physical verification standard regardless of host type.
This is not the detection layer performing consistently---it is the verification
layer.

\subsection{Case Studies}

\textbf{T1071.001 --- Active C2 web protocol (nfury, refuted).}
Memory analysis flagged T1071.001 from the string \texttt{established} in
\texttt{windows.netscan} output. TCP connection state strings are present in
memory regardless of active malicious connections. A prompt-guardrailed system
would flag this technique with a caveat. The Forensic Auditor ran
\texttt{windows.netscan} and checked all 432 connection records. Every
\texttt{ESTABLISHED} and \texttt{CLOSE\_WAIT} connection resolved to Apple,
Microsoft, or Google CDN infrastructure. Verdict: \textsc{refuted}. The Auditor
did not conclude \textsc{refuted} because it was instructed to be skeptical. It
concluded \textsc{refuted} because it ran out of connections to check and found
none that supported the claim.

\textbf{T1547.001 --- Registry Run Keys (nfury, refuted).}
Memory analysis flagged T1547.001 from the string \texttt{currentversion\textbackslash run}
appearing in memory. Over three challenge rounds, the Auditor checked the
\texttt{SOFTWARE} hive, \texttt{SAM}, \texttt{SECURITY}, \texttt{BCD}, and
\texttt{Syscache} system hives, plus 11 \texttt{NTUSER.DAT} files across all
user profiles. Volatility \texttt{windows.registry.printkey} against
\texttt{HKLM\textbackslash SOFTWARE\textbackslash Microsoft\textbackslash
Windows\textbackslash CurrentVersion\textbackslash Run} returned two entries:
\texttt{Sidebar} $\to$ \texttt{\%ProgramFiles\%\textbackslash Windows
Sidebar\textbackslash Sidebar.exe /autoRun} under \texttt{NetworkService} and
\texttt{LocalService}, both dated 2009-07-14---the stock Windows~7 installation
timestamp. The Auditor's conclusion: ``the triage signal appears to have been
triggered by a string match in memory from the OS loader or registry paging, not
by an actual malicious value stored in the key.'' Verdict: \textsc{refuted}.

\textbf{T1055 --- Process injection (rocba, confirmed under resource pressure).}
rocba is a C2 relay node with zero disk artifacts by design. All five candidate
techniques came from memory analysis. In Round~1, \texttt{windows.malfind}
timed out after 120~seconds; the Auditor returned \textsc{inconclusive}---not
\textsc{confirmed}. The system failed safe. In Round~2, the Auditor recovered
one VAD record before the timeout: \texttt{MsMpEng.exe} (Windows Defender's
engine) with two anonymous \texttt{VadS} \texttt{PAGE\_EXECUTE\_READWRITE}
regions at \texttt{0x1f081330000} and \texttt{0x1f0818a0000}, containing an
x64 shellcode dispatch trampoline (\texttt{push rsi/rdi/rbx; sub rsp, 0x28;
jmp rdx}). Verdict: \textsc{confirmed}. A hallucination-prone system would have
confirmed T1055 in Round~1 from the timeout result or from the memory triage
signal. The Auditor returned \textsc{inconclusive} until it had physical bytes.

\subsection{Cross-Host Campaign Correlation}

Running tdungan in campaign mode with nfury IOCs injected, the Auditor confirmed
that the \texttt{SRL-Helpdesk} account NTLM hash (\texttt{4c3f5e9f...}) matched
exactly across both hosts. This is confirmed credential reuse from a physical
artifact---the hash extracted from memory by Volatility \texttt{windows.hashdump}
on both hosts---not from model inference about campaign patterns. The IOC
injection mechanism (\cref{sec:ioc}) passes the verified hash as a structured
signal, not as LLM context.

%% ── 6. Discussion ────────────────────────────────────────────────────────────
\section{Discussion}
\label{sec:discussion}

\textbf{Categorical vs.\ probabilistic safety guarantees.}
Bensalem et al.~\cite{bensalem2026} derive compositional system-level safety
bounds via the chain rule of probability, producing probabilistic guarantees for
each layer of their architecture. For Layer~2 (environmental validity), their
framework produces a bound $P(\text{environmentally valid} \mid \text{proposed})$.
The Forensic Auditor enforces environmental validity categorically: by data flow
construction, \textsc{confirmed} cannot be issued without a positive forensic
tool return. No probability estimate is needed because the property holds
unconditionally. This is a stronger guarantee than the probabilistic framework
admits, achievable in the forensic domain because the verification oracle
(the filesystem) is deterministic.

\textbf{Separation from AgentVerify.}
AgentVerify~\cite{fang2026} achieves 86.67\% post-hoc verification accuracy on
designed agent scenarios using LTL model checking over observable control flow.
VERITAS achieves a proactive structural guarantee rather than post-hoc detection.
The two approaches are complementary: AgentVerify can verify whether VERITAS's
own control flow satisfies its safety specification; VERITAS prevents the
specific failure mode (hallucinated confirmation) before it occurs.

\textbf{Forensic admissibility.}
AI-generated evidence faces an emerging legal challenge: if the model's reasoning
cannot be inspected or reproduced, a finding may be excludable on
due-process grounds~\cite{aronson2010}. Every VERITAS \textsc{confirmed} finding
consists of an exact command, an exact output, and an exact UTC timestamp in an
append-only log. A second examiner can mount the same image, run the same command,
and reach the same conclusion without the AI. The AI decided to run the command.
The output is the evidence. This architecture is designed for the legal scrutiny
that is coming for forensic AI.

\textbf{The refutals as primary evidence.}
It is tempting to read the 32 confirmed findings as the system's primary output
and the 16 refutals as noise. This is backwards. The confirmed findings
demonstrate that the system finds things. The refuted findings demonstrate that
the system distinguishes between a memory signal and a physical artifact---the
property the architecture was designed to enforce. A system that refutes nothing
is a system that never challenges its own triage output. The exactly-four pattern
across all four hosts is the paper's central empirical result.

%% ── 7. Limitations ───────────────────────────────────────────────────────────
\section{Limitations}
\label{sec:limitations}

\textbf{Four hosts, one campaign.} All four evaluated hosts share an operator,
C2 infrastructure (192.168.1.5), and overlapping tooling (httppump/spinlock.exe).
The consistent Auditor behavior is encouraging but does not constitute a
generalization test. Running against a second campaign with a different threat
actor, different tooling, and no prior IOC seeding is the immediate next
validation step.

\textbf{Triage layer is a proof of concept.} The corpus-calibrated log-odds
weights cover 9 MITRE techniques. The 87\% false positive rate on benign
endpoints disqualifies the triage layer for live endpoint monitoring. It is
calibrated for forensic images pre-selected for analysis where the base rate of
compromise is high. The Auditor is the precision layer.

\textbf{Sysmon ML does not generalize to disk forensics.} An adversarial
Red/Blue training loop (3,000 iterations, 49,519 Sysmon events) produced
hardened detection rules. Those rules operate on live Sysmon telemetry fields
(\texttt{ProcessGuid}, \texttt{CommandLine}, \texttt{ParentImage}) that are
absent from static disk forensic output. The training infrastructure is
operational; the deployment context for the current pipeline is wrong. This
layer is documented as future work for a live endpoint path.

\textbf{Single-model adversarial ceiling.} The Red and Blue agents in the
training loop share the same underlying LLM. Claude cannot simulate a
zero-day technique it has not encountered in training data. The evasion variants
it generates are plausible mutations of known techniques, not novel attacks.

\textbf{Cost and scale.} \$14 per host and 16--23 minutes per investigation is
appropriate for targeted triage of high-value hosts in an active incident. It
does not scale to continuous monitoring of 500 simultaneous endpoints without
parallel infrastructure and pre-staged images.

\textbf{T1574 triage signal quality.} The triage signal for T1574 (Hijack
Execution Flow) on nfury was the literal string \texttt{false\textbackslash t}---
a parser artifact from a null boolean field. The Auditor independently identified
this as anomalous signal quality and refuted the finding. This demonstrates
architectural robustness to poor-quality triage input but also reveals that the
triage layer's signal extraction can produce garbage signals that reach the
Auditor.

%% ── 8. Future Work ───────────────────────────────────────────────────────────
\section{Future Work}
\label{sec:future}

\textbf{Novel campaign validation.} The immediate priority is running the current
pipeline against a second campaign with different tooling and no prior IOC
seeding, measuring false positive rate, missed techniques, and Auditor correction
rate independently.

\textbf{Neural network triage with Auditor as admissibility layer.} A deep
model trained on real Sysmon telemetry with BETH dataset pool-separation will
outperform log-odds ratios on detection rate. NN weights cannot be explained in
a chain-of-custody review. The Auditor is the answer: any triage signal---corpus
weight, neural network, YARA rule---feeds the same verification layer.
\textsc{confirmed} remains a tool return value regardless of what generated
the candidate. The NN provides detection rate. The Auditor provides admissibility.

\textbf{Live Sysmon endpoint path.} The adversarial training loop was built on
the correct telemetry format. Connecting it to a live MCP endpoint where Sysmon
events actually flow is where the trained rules become a production detection
layer.

\textbf{Plaso timeline correlation.} Filtering a supertimeline to the window
around each confirmed technique timestamp turns isolated artifact matches into
activity chains---the difference between ``this binary existed on disk'' and
``this binary ran at 14:32:07, four minutes before this network connection.''

\textbf{Coverage expansion.} Nine MITRE techniques are covered by corpus weights.
The MalwareBazaar and HybridAnalysis APIs scale this to 50+ with additional
corpus collection.

%% ── 9. Conclusion ────────────────────────────────────────────────────────────
\section{Conclusion}

The hallucination problem in autonomous DFIR is architectural, not
prompt-engineering. Separating the finding-proposal agent from the
finding-verification agent---with zero shared state, a mandate to refute, and
a requirement that \textsc{confirmed} demands a positive forensic tool
return---makes hallucinated confirmation structurally impossible. This property
holds by data-flow construction and is verifiable by reading the code.

Across four hosts spanning two tool families and both victim and C2
infrastructure, the Forensic Auditor refuted exactly four techniques per host
while confirming thirty-two from forty-eight detected. Every confirmed finding
traces to a specific tool call in an append-only audit log and is independently
reproducible without the AI. The human examiner reviews artifact-backed
verdicts, not LLM output.

Valhuntir built the investigation layer and left the verification problem open.
VERITAS adds the verification layer. The architecture is designed to survive both
the forensic scrutiny that the field demands and the legal scrutiny that is coming
for AI-generated evidence.
